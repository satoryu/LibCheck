# Issue #25: 検索履歴の保存 — 要件定義

## Problem Statement

現在、蔵書検索を行った結果はアプリ内で一時的に表示されるのみで、画面遷移すると失われる。ユーザーは過去に検索した書籍を再度確認したい場合、ISBNを再入力して検索し直す必要がある。検索結果をローカルストレージに永続化し、後から参照・再検索できる基盤を構築する。

## Requirements

### Functional Requirements

1. **自動保存**: 蔵書検索が正常に完了した際、検索履歴をローカルストレージに自動的に保存する
2. **保存データ**: 以下の情報を保存する
   - ISBN（13桁）
   - 検索日時（ISO 8601形式）
   - 各図書館の蔵書状況（systemId → AvailabilityStatus のマッピング）
3. **永続化**: アプリ再起動後も保存データを参照可能にする
4. **同一ISBN更新**: 同一ISBNで再検索した場合、既存の履歴レコードを最新の結果で上書きする（重複を作らない）
5. **最大件数制限**: 履歴の最大保存件数を100件とし、上限を超えた場合は最も古い履歴から自動削除する
6. **履歴取得**: 保存された検索履歴を検索日時の降順（新しい順）で取得できる
7. **個別削除**: 指定したISBNの履歴を削除できる
8. **全件削除**: すべての検索履歴を一括削除できる

### Non-Functional Requirements

1. **パフォーマンス**: 履歴の保存・取得は100件以内で100ms未満で完了すること
2. **データ整合性**: アプリの異常終了時もデータが破損しないこと（SharedPreferencesのアトミック書き込みに依存）
3. **テスタビリティ**: リポジトリパターンを使用し、ユニットテストが容易な設計にすること

## Constraints

- ローカルストレージとして既存の `shared_preferences` パッケージを使用する
- 既存の Clean Architecture レイヤー構成（domain / data / presentation）に従う
- 既存の Riverpod による状態管理パターンに従う

## Acceptance Criteria

1. 蔵書検索が完了すると、検索履歴が自動的にローカルストレージに保存される
2. アプリを再起動しても保存された検索履歴が読み込める
3. 同一ISBNで再検索すると、既存の履歴が更新され重複レコードが作られない
4. 履歴が100件を超えると、最も古い履歴が自動的に削除される
5. 個別削除・全件削除が正常に動作する
6. すべてのドメインロジック・リポジトリにユニットテストが存在する

## User Stories

1. **検索結果の自動保存**: ユーザーとして、蔵書検索を行うと結果が自動的に保存されるので、後から過去の検索を参照できる
2. **再検索時の更新**: ユーザーとして、同じ本を再検索すると最新の結果に更新されるので、常に最新の蔵書状況を確認できる
3. **履歴の削除**: ユーザーとして、不要な履歴を削除できるので、履歴リストを整理できる
