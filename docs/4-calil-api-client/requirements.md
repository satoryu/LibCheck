# Issue #4: カーリルAPIクライアントの実装 — 要件定義

## 問題定義

LibCheckアプリでは、ユーザーが登録した図書館の蔵書検索を行うために、外部APIである[カーリル図書館API](https://calil.jp/doc/api_ref.html)との通信が必要である。現時点ではAPIクライアントの基盤が存在しないため、図書館検索および蔵書検索の機能を実装できない。

本Issueでは、カーリルAPIとの通信を担うクライアント基盤を構築し、以降のPhase（図書館登録・蔵書検索）で利用可能な状態にする。

## 機能要件

### FR-1: 図書館検索API (`/library`)

- **FR-1.1**: 都道府県名を指定して図書館一覧を取得できること
- **FR-1.2**: 都道府県名 + 市区町村名を指定して図書館一覧を絞り込みできること
- **FR-1.3**: レスポンスから図書館情報（名称、住所、システムID、図書館キーなど）をドメインモデルに変換できること
- **FR-1.4**: JSON形式でレスポンスを受け取れること（`format=json`パラメータ指定）

### FR-2: 蔵書検索API (`/check`)

- **FR-2.1**: ISBN（1冊〜複数冊）とシステムID（1つ〜複数）を指定して蔵書検索をリクエストできること
- **FR-2.2**: ポーリング機構を実装し、`continue=1`の間は2秒以上の間隔を空けて再リクエストすること
- **FR-2.3**: ポーリング完了時（`continue=0`）に最終結果をドメインモデルとして返却すること
- **FR-2.4**: 各図書館の蔵書状態（貸出可・貸出中・蔵書なし等）を判定できること
- **FR-2.5**: 予約ページURL（`reserveurl`）を取得できること

### FR-3: APIキー管理

- **FR-3.1**: APIキーをアプリ内で安全に管理できること（ハードコードしない）
- **FR-3.2**: APIキーを外部設定（環境変数や設定ファイル）から読み込めること

## 非機能要件

### NFR-1: パフォーマンス

- **NFR-1.1**: APIレスポンスのタイムアウトは10秒とする
- **NFR-1.2**: ポーリング間隔は最低2秒（API仕様に準拠）
- **NFR-1.3**: 蔵書検索の最大ポーリング回数を設定可能とする（デフォルト: 30回 = 最大約60秒）

### NFR-2: 信頼性

- **NFR-2.1**: ネットワークエラー時に適切な例外を発生させること
- **NFR-2.2**: APIエラーレスポンス（status: Error）を適切にハンドリングすること
- **NFR-2.3**: 不正なJSONレスポンスに対して適切なエラーハンドリングを行うこと

### NFR-3: テスタビリティ

- **NFR-3.1**: HTTPクライアントを抽象化し、テスト時にモックに差し替え可能とすること
- **NFR-3.2**: APIクライアントの全メソッドに対してユニットテストを作成すること

### NFR-4: レート制限

- **NFR-4.1**: APIのレート制限（1,000リクエスト/時/IP・APIキー）を考慮した設計とすること
- **NFR-4.2**: 蔵書検索のISBN同時指定は最大100件とすること

## 制約

- **C-1**: HTTPパッケージとして `http` を使用する（pubspec.yamlに追加）
- **C-2**: Clean Architectureのレイヤー分離に従い、ドメインモデルは `lib/domain/`、APIクライアントは `lib/data/` に配置する
- **C-3**: 状態管理には Riverpod を使用する（Phase 1で導入済み）
- **C-4**: カーリルAPIのアプリケーションキー登録が必要（https://calil.jp/api/dashboard/）
- **C-5**: APIレスポンスはJSON形式を使用する

## 受入基準

- [ ] `CalilApiClient` クラスが実装され、図書館検索・蔵書検索の2つのメソッドを持つこと
- [ ] 図書館検索メソッドが都道府県名（+オプションで市区町村名）を引数に取り、`Library` モデルのリストを返すこと
- [ ] 蔵書検索メソッドがISBNリストとシステムIDリストを引数に取り、ポーリングを行い、`BookAvailability` モデルを返すこと
- [ ] ポーリングが2秒以上の間隔を守っていること
- [ ] ドメインモデル（`Library`, `BookAvailability`, `LibraryStatus` 等）が定義されていること
- [ ] HTTPクライアントがDIで差し替え可能であること
- [ ] 全メソッドにユニットテストが存在し、パスすること
- [ ] エラーケース（ネットワークエラー、APIエラー、タイムアウト）のテストが存在すること

## ユーザーストーリー

### US-1: 図書館一覧の取得

> 開発者として、都道府県名を指定してカーリルAPIから図書館一覧を取得したい。
> それにより、Phase 2の図書館登録UIで図書館リストを表示できるようになる。

### US-2: 蔵書検索の実行

> 開発者として、ISBNと図書館システムIDを指定してカーリルAPIで蔵書検索を実行したい。
> それにより、Phase 3の蔵書検索機能で各図書館の貸出状況を表示できるようになる。

### US-3: ポーリングによる結果取得

> 開発者として、蔵書検索APIのポーリング機構を意識せずに最終結果を取得したい。
> それにより、上位レイヤーはポーリングの詳細を気にせず蔵書状況を利用できるようになる。

### US-4: テストでのモック差し替え

> 開発者として、APIクライアントのHTTP通信部分をモックに差し替えたい。
> それにより、ネットワークに依存しないユニットテストを作成できるようになる。
